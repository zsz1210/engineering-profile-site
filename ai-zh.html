How I used AI

AI的使用手法與型態

我會先在人類側決定目的、限制條件與完成條件，再把調查、設計案、實作、程式審查、 測試觀點與文件化分別交給AI支援。

使用核心是狀態管理。使用環境依序從Gemini、OpenClaw、Hermes Agent演進到Codex， 同時透過GitHub、本機檔案、瀏覽器確認、記憶資訊與驗證結果，讓作業能跨Session持續。

我不會直接採用AI輸出，而是把差分確認、測試、靜態分析、畫面確認、Log確認、 程式審查與發布準備都放進完成條件。

涉及公司資訊、個人資訊、Token、外部傳送、破壞性操作與發布判斷時， 我會把AI可協助的範圍與人類需要確認的範圍分開。

以Mac mini為中心的AI開發環境

我將Mac mini作為AI開發與自動化的本機基盤。即使使用環境從Gemini、OpenClaw、 Hermes Agent演進到Codex，我也持續連接GitHub、瀏覽器、文件與記憶資訊。

Codex in daily work

將Codex作為實作與驗證的整合環境

在寫程式前，我會先用Codex確認本機記憶、GitHub履歷與目前差分。 接著在同一個流程中完成檔案編輯、瀏覽器確認、成果物製作與驗證。

codex session

先確認「這是哪個作業的延續」

我不會直接進入實作，而是先確認本機記憶、PR、Issue、目前差分與release branch。 尤其是接手既有作業時，不只看main就做判斷。

Local files 閱讀履歷、網站、設定與專案檔案，反映到正確成果物
Browser check 實際開啟本機網站或Web App，確認版面、導線與使用者可見狀態
GitHub workflow 從Issue、PR、branch與Project履歷復原作業狀態
Verification 把測試、差分確認、審查、畫面確認與Log確認放入完成條件

How I ran the work

以GitHub為共同狀態，用短週期推進

GitHub不只是放程式碼的地方，而是人類與AI能看到同一個作業狀態的中心。

  1. Issue

    閱讀目的與限制

    從Issue、Project、既有PR與本機記憶復原這次作業屬於哪個流程。

  2. Branch

    選擇正確的延續點

    不只看main，而是確認release branch與過去PR再決定開始位置。

  3. Build

    小步實作與驗證

    閱讀差分、執行測試、分析警告、審查輸出與確認畫面，讓AI同時支援實作與確認。

  4. Deploy

    連接發布與驗證記錄

    將建置上傳、發布準備、PR、Issue留言與驗證Log接在一起。

  5. Observe

    用AI輔助SLO與監控

    整理Metrics、Log、Alert與SLO狀態，快速判斷異常徵兆與影響範圍。

  6. Incident

    自動化正式環境問題的初動

    彙整相關Log、近期發布、監控資訊與既知Issue，支援一次切分與對應候選。

AI governance / security

在公司資訊前提下設計AI使用邊界

AI活用不只要展現速度，也需要說明什麼不能交給AI、哪些地方需要人類確認。

不記錄秘密資訊

API key、GitHub Token、內部ID與個人資訊不放進公開資料或公開Log。

外部傳送需要確認

發文、PR操作、檔案共享、破壞性操作與公開判斷都需要人類確認。

公開說明時抽象化

公開網站與履歷中，將公司名、內部構成與具體設定值做抽象化與遮蔽。

分離本機權限

以Mac mini作為執行基盤時，分開處理強本機權限與外部整合範圍。

Future perspective

我對AI活用未來的看法

AI會從單純寫程式的工具，擴展到調查、設計、實作、驗證、營運、監控與決策支援。 重要的不是導入AI本身，而是把AI安全地設計進企業流程、開發團隊與資訊管理。

Industry

對產業的影響

競爭力會從「能不能使用AI」轉向「能不能以AI為前提重新設計業務流程」。

Company

對企業的影響

企業需要設計權限、資料邊界、審查、稽核、安全性與發布判斷規則。

Engineer

對工程師的影響

工程師不再只看寫了多少程式，而是更重視課題分解、脈絡設計、審查能力與風險判斷。

00 / Product origin

merical是管理二手平台銷售利潤、庫存與銷售額的手機App

merical用來管理仕入價格、預定售價、平台手續費、運費、包材、銷售額與利潤。 它涵蓋從仕入到售出的紀錄、利潤模擬、包材庫存、銷售分析與備份。

  • Flutter / iOS App
  • 利潤計算、仕入、庫存、銷售管理
  • Drift / SQLite本機資料庫
  • 備份、廣告、Pro課金導線

01 / From zero to release

merical是從零開始活用AI製作的產品

我從業務課題整理、需求定義、功能切分、畫面構成、資料模型與架構方針開始使用AI。 接著推進實作、測試、解析、模擬器確認、實機確認、TestFlight確認、建置上傳與發布準備。

  • 從需求定義開始AI輔助
  • Feature-First / Clean Architecture
  • 從模擬器到實機皆進行驗證
  • TestFlight確認與發布準備皆有實施

02 / Agile AI team operation

在merical中，我讓AI擔任不同角色，如同小型開發團隊運作

我自己負責產品判斷與最終確認，AI端則分成進度管理、實作、驗證、程式審查與發布準備。 以GitHub Project、release branch、PR與TestFlight確認作為共同循環。

Lead 進度管理

整理Issue、Project狀態、release branch與下一個作業。

Builder 實作

閱讀Flutter/iOS差分，支援功能修正、設定變更與建置調整。

Tester 驗證

管理測試、解析、模擬器、實機與TestFlight的確認觀點。

Reviewer 品質確認

確認PR差分、既有警告、新風險與驗證Log是否不足。

Release 發布準備

整理IPA建置、上傳、App Store Connect狀態與發布判斷材料。
