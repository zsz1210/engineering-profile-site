How I used AI

Methods and Operating Style

I first define the objective, constraints, and completion criteria on the human side. Then I ask AI to support research, design options, implementation, review, test viewpoints, and documentation as separate tasks.

The core is state management. I connect Codex, Hermes Agent, OpenClaw, Gemini, GitHub, local files, browser checks, memory, and verification results so work can continue across sessions.

I do not accept AI output as-is. Diff checks, tests, static analysis, UI checks, logs, reviews, and release preparation are part of the completion criteria.

I also separate what AI can handle from what requires human confirmation, especially around company data, personal data, tokens, external transmission, destructive actions, and release decisions.

Codex in daily work

Before writing code, I use Codex to inspect local memory, GitHub history, and current file diffs. Then I use it for file edits, browser checks, document creation, and verification.

How I ran the work

Run Short Cycles with GitHub as the Shared State

I used GitHub not only as a code repository, but as the center where humans and AI agents share the current state.

  1. Issue

    Read purpose and constraints

    Use issues, projects, existing PRs, and local memory to understand what flow this work belongs to.

  2. Branch

    Choose the correct continuation point

    Do not assume main is correct. Check release branches and previous PRs before starting.

  3. Build

    Implement and verify in small steps

    Read diffs, run tests, analyze warnings, review output, and check screens while using AI as a support role.

  4. Deploy

    Connect deployment and verification logs

    Keep build uploads, release preparation, PRs, issue comments, and verification logs connected.

  5. Observe

    Use AI for SLO and monitoring support

    Summarize metrics, logs, alerts, SLO status, early signals, and impact scope for faster judgment.

  6. Incident

    Automate the first response to production issues

    Gather related logs, recent releases, monitoring information, and known issues to support initial triage.

AI governance / security

Designing Guardrails for AI Use with Company Information

AI adoption should show not only speed, but also what should not be sent to AI and where human confirmation is required.

Do not record secrets

Do not include API keys, GitHub tokens, internal IDs, or personal information in public materials.

Confirm external transmission

Require human confirmation for posts, PR operations, file sharing, destructive actions, and publication decisions.

Sanitize for public explanation

Abstract company names, internal architecture, and specific settings when explaining work publicly.

Separate local permissions

Separate strong local permissions from external integrations when using a Mac mini as the execution base.

Future perspective

How I See AI Evolving

AI will expand from code generation into research, design, implementation, verification, operations, monitoring, and decision support. The important part is not introducing AI itself, but designing it safely into company workflows, development teams, and information management.

Industry

Impact on the industry

Competitiveness will shift from whether a company uses AI to whether it can redesign business processes around AI.

Company

Impact on companies

Companies need rules for authority, data boundaries, review, auditability, security, and release decisions.

Engineer

Impact on engineers

Engineers will be evaluated more by problem decomposition, context design, review ability, and risk judgment than by code volume alone.
