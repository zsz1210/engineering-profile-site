How I used AI

AIをどんな手法・形で利用していたか

AIは、単発の質問相手としてではなく、開発プロセスの中に役割を持たせて使っていました。 まず人間側で目的、制約、完了条件を決め、そのうえでAIに調査、設計案、実装、レビュー、 テスト観点、ドキュメント化を分けて依頼します。

使い方の中心は、状態管理です。Codex、Hermes Agent、OpenClaw、Geminiを用途で分け、 GitHub、ローカルファイル、ブラウザ、記憶情報、検証結果をつなげることで、 セッションが変わっても同じ作業を継続できるようにしていました。

もう一つの軸は検証です。AIにコードを書かせるだけではなく、差分確認、テスト、画面確認、 ログ確認、レビュー、リリース準備までを完了条件に入れます。 そのため、AI活用はスピードだけでなく、品質と引き継ぎ可能性を上げるための仕組みとして扱っています。

さらに、AIを使う範囲と、人間が確認する範囲を分けています。 会社情報や個人情報、Token、外部送信が絡む場面では、AIへ判断を委ねきらず、確認やサニタイズを挟むようにしていました。

How I think

実際に意識していたこと

AI活用で大きなリスクになるのは、便利さの裏で作業状態が散らかることです。 そのため、会話、ファイル、GitHub、検証結果をつなげて、次に続けられる形にすることを意識していました。

01

会話より状態を信じる

チャットログだけに依存せず、現在のタスク、判断、検証結果、次のアクションをファイルやGitHubへ残します。 セッションが変わっても同じ文脈に戻れることを重視しています。

02

AIに任せる範囲を分ける

調査、差分確認、実装、テスト補助はAIに寄せつつ、外部送信、破壊的操作、公開判断、 リリース判断は人間確認を挟む設計にしています。

03

失敗を次の運用に変換する

繰り返したミス、環境依存の詰まり、確認漏れは、手順やルールに戻して次回の作業品質を上げます。 AI利用そのものも改善対象として扱います。

How I ran the work

GitHubを見ながら、短いサイクルで回す

GitHubは、単なるコード置き場ではなく、AIと人間が同じ状態を見て作業するための中心にしていました。 Issue、PR、Project、branch、検証ログを見ながら、AIに役割を持たせて短いサイクルで進めます。

  1. Issue

    目的と制約を読む

    Issue、Project、既存PR、ローカル記憶から、今回の作業がどの流れの継続かを復元します。

  2. Branch

    正しい継続地点を選ぶ

    mainだけを見て判断せず、release branchや過去PRを確認して作業開始地点を決めます。

  3. Build

    実装と検証を小さく回す

    差分を読み、テスト、解析、レビュー、画面確認を挟みながら、AIに実装と確認の両方を補助させます。

  4. Verify

    検証結果を判断材料にする

    テスト結果、画面確認、ログ、レビュー指摘を残し、次に進めるかどうかを判断できる状態にします。

  5. Deploy

    デプロイと検証ログを接続する

    ビルドアップロード、リリース準備、PR、Issueコメント、検証ログをつなぎ、引き継ぎ可能な状態にします。

  6. Observe

    SLOと監視をAIで補助する

    メトリクス、ログ、アラート、SLOの状態をAIで整理し、異常の兆候、影響範囲、優先度を素早く判断できるようにします。

  7. Incident

    本番不具合の初動を自動化する

    本番障害時に、関連ログ、直近リリース、監視情報、既知Issueを集約し、一次切り分けと対応候補を提示する流れを作ります。

AI governance / security

会社情報が絡む前提で、AI利用の守りも設計する

AI活用のアピールでは、スピードだけでなく「何をAIに渡さないか」「どこで人間確認するか」も重要です。 業務で使えるAI体制として、情報保護と外部連携の境界を明確にします。

秘密情報を記録しない

APIキー、GitHub Token、Discord Token、内部ID、個人情報を資料や公開ログに残さない。

外部送信は確認する

チャット投稿、PR操作、ファイル共有、破壊的操作、公開判断は人間確認を挟む。

公開用に抽象化する

サイトや職務経歴書では、会社名、内部構成、具体的な設定値をサニタイズして説明する。

ローカル権限を分離する

Mac mini上の実行基盤では、強いローカル権限と外部連携の範囲を分けて扱う。

Future perspective

AI活用の今後

AIは、単にコードを書く道具から、調査、設計、実装、検証、運用、意思決定の補助まで広がっていくと考えています。 重要なのは、AIそのものを導入することではなく、企業の業務フロー、開発チーム、情報管理の中で安全に使える形へ設計することです。

Industry

業界への影響

AIは、ソフトウェア開発だけでなく、業務整理、顧客対応、データ分析、ドキュメント作成、運用監視へ広がっていきます。 今後は「AIを使えるか」よりも、「AIを前提に業務プロセスを再設計できるか」が競争力になります。

Company

企業への影響

企業では、AIを試す段階から、権限、データ境界、レビュー、監査、セキュリティを含めた運用設計が必要になります。 小さなPoCで終わらせず、現場の作業ログ、GitHub、ドキュメント、リリース判断と接続することが重要です。

Engineer

エンジニアへの影響

エンジニアの役割は、コードを書く量だけでは測れなくなります。 課題を分解する力、AIへ渡す文脈を設計する力、出力をレビューする力、品質とリスクを判断する力がより重要になります。

今後の発展イメージ

AI for SRE / Operations

SLO・監視・本番障害対応へのAI活用

AIは開発工程だけではなく、運用と信頼性改善にも使えると考えています。 特に本番不具合では、最初の数分で何を見て、誰に伝え、どこまで自動化するかが重要です。

01

SLO / Error Budgetの整理

サービスごとのSLO、エラーバジェット、直近の劣化傾向をAIに整理させ、対応優先度を判断しやすくする。

02

監視アラートの一次分析

Datadogなどのメトリクス、ログ、トレース、アラート文面を集約し、関連しそうな原因候補を短時間で出す。

03

本番不具合の初動支援

直近リリース、PR、設定変更、既知Issue、エラーログを見て、影響範囲、暫定対応、切り戻し判断材料をまとめる。

04

障害後の分析と再発防止

タイムライン、原因、検知遅れ、対応遅れ、改善タスクを整理し、Postmortemと次のSLO改善へつなげる。

Operating model

AI時代に必要だと思う開発体制

AIを導入するだけでは、企業の成果にはつながりません。 誰が目的を決めるのか、どのデータを使えるのか、AIの出力を誰がレビューするのか、 どの基準を満たしたらリリースできるのか、本番障害時にどこまで自動対応してよいのかを決める必要があります。

そのため、今後はエンジニアがAIを使ってコードを書く力だけでなく、 AIを含む開発・運用プロセスそのものを設計し、チームが安全に速く動ける状態を作る力が重要になると考えています。

00 / Product origin

mericalとは、フリマ販売の利益・在庫・売上を管理するモバイルアプリ

mericalは、メルカリなどのフリマ販売で発生する「仕入れ価格」「販売予定価格」 「販売手数料」「送料」「梱包資材」「売上」「利益」を一元管理するためのアプリです。 仕入れから売却までの記録、利益シミュレーション、資材在庫、売上分析、バックアップまでを扱います。

  • Flutter / iOSアプリ
  • 利益計算、仕入れ、在庫、売上管理
  • Drift / SQLiteによるローカルDB
  • バックアップ、広告、Pro課金導線

01 / From zero to release

mericalは0からAIを活用して作ったプロダクト

まず業務課題を整理し、要件定義、機能分割、画面構成、データモデル、アーキテクチャ方針をAIと検討しました。 その後、Flutter / Riverpod / Driftを使って実装し、テスト、解析、シミュレーター確認、実機確認、 TestFlight確認、ビルドアップロード、デプロイ準備までを一連の開発サイクルとして進めました。

  • 要件定義からAI活用
  • Feature-First / Clean Architecture
  • シミュレーターから実機まで検証
  • TestFlightとデプロイ準備まで実施

What happened in the project

mericalでの経緯: 課題の発見から、AI前提の開発体制へ

1. フリマ販売の手間をプロダクト課題として整理

仕入れ価格、販売価格、送料、手数料、梱包資材、売上記録が別々に散らばると、 実際の利益が見えにくくなります。そこを、個人のフリマ販売向けの管理アプリとしてまとめる方針にしました。

2. 要件定義と設計からAIを使う

利益シミュレーター、仕入れ管理、資材在庫、売上分析、設定、バックアップを機能単位で分け、 Feature-FirstとClean Architectureを前提に、長く拡張できる構成をAIを使って検討しました。

3. 実装はAIに役割を持たせて進める

実装、レビュー、テスト、リリース確認を分け、AIに期待する出力を明確にしました。 画面、状態管理、DB、バックアップ、広告、課金導線を小さな単位に分けて進めることで、作業を継続しやすくしました。

4. シミュレーターだけで終わらせない

Flutterのテストやシミュレーター確認だけではなく、実機、iPhone Mirroring、TestFlightまで確認対象にしました。 特にバックアップやiCloud、課金・広告まわりは、実機でしか見えない差分があるため、AIに検証観点を整理させながら進めました。

5. リリース段階では履歴とGitHubを使って継続

後半では、Hermes Agentの作業履歴、GitHubのPR、release branch、Project boardを見て、 どの状態から再開すべきかを整理しました。IPAビルド、アップロード、App Store Connectの処理状況、 TestFlightでの確認結果も作業ログとして残しました。

02 / Agile AI team operation

mericalではAIに役割を持たせ、小さな開発チームとして運用

私自身がプロダクト判断と最終確認を担い、AI側には進行管理、実装、検証、レビュー、 リリース準備の役割を分けました。GitHub ProjectのBacklog / In progress / In reviewと release branchを起点に、短いサイクルで実装、検証、PR、TestFlight確認を回す形です。

Lead 進行管理

Issue、Project、release branchから現在地を整理し、次に進める作業へ分解。

Builder 実装

Flutter/iOSの差分を読み、機能修正、設定変更、ビルド調整を担当。

Tester 検証

テスト、解析、シミュレーター、実機、TestFlightの確認観点を管理。

Reviewer 品質確認

PR差分、既存警告、新規リスク、検証ログの不足を確認。

Release デプロイ準備

IPAビルド、アップロード、App Store Connect処理、リリース判断材料を整理。

  1. Backlog / Issueを確認
  2. 作業を小さく分割
  3. 実装と検証を反復
  4. PRとProjectを更新
  5. 実機・TestFlightで確認
  6. 学びを次の運用へ反映
